# ボケトレ！ 実装準備仕様書

## 1. 製品定義

「ボケトレ！」は、人以外の森羅万象を教材に、音声回答と反復練習で発想力を鍛えるスマートフォン向け大喜利トレーニングアプリである。

主目的は採点そのものではなく、次の学習ループを成立させること。

1. 撮影、フリー画像、オフライン生成画像から教材を選ぶ
2. お題に音声で回答する
3. 10種類の評価軸で100点満点の分析を受ける
4. 最も弱い能力に対する改善ヒントを受ける
5. 同じ教材・同じお題へ再回答する
6. 前回からの変化を確認する

## 2. 確定したプロダクト判断

- スマートフォン縦持ちを第一対象とする。
- 人物は撮影対象外とする。
- コーチに実在人物の完全な名前、肖像、音声を使用しない。
- コーチ名は伏字表記とし、正式な機能名は評価軸とする。
- 現在の表示名は `松本◯志 / 有吉◯行 / 粗◯ / 赤嶺総◯ / 真空ジェシカ・川北◯澄 / 設楽◯ / 寺田◯明 / 千原ジュ◯ア / バカリ◯ム / 笑い飯・西田◯治`。
- 伏字と実在人物の対応表は製品内に保存しない。
- 本人の再現率を保証しない。説明可能な評価タイプとして提供する。
- フリー画像はOpenverseから都度取得し、アプリ資産として保存しない。
- オフライン時は端末上で練習画像を生成する。
- 写真と音声は原則として外部送信・永続保存しない。
- 採点根拠をユーザーが確認できるようにする。
- PWAを最初の製品版とし、その後ネイティブラッパーを検討する。

## 3. MVP機能

### 教材

- 背面・前面カメラの切り替え
- 正方形撮影
- 人物撮影禁止表示
- 対応端末での顔検出
- OpenverseからのCC0・PDM画像取得
- 作品名、作者、ライセンス、原典の表示
- 通信失敗時の端末生成画像

### 回答

- 日本語音声認識
- 15秒制限
- 中間認識結果の表示
- 音声認識非対応時の明確な案内
- 回答失敗・無回答からの復帰

### トレーニング

- 20種類以上のお題
- 10種類の評価軸
- 100点満点の分析
- 特徴量と採点根拠の表示
- 弱点別改善ヒント
- 同じ問題への再挑戦
- 再挑戦前後の得点差
- セッション内回答数
- ベストスコア

### オフライン

- アプリシェルのキャッシュ
- お題と採点のオフライン実行
- 端末生成教材
- 最後に取得した採点設定の利用

## 4. MVP対象外

- 人物を含む写真の利用
- 実在人物本人の音声・肖像・発言再現
- 公開ランキング
- SNS投稿
- 他ユーザーとの対戦
- 生の写真・音声のクラウド保存
- 自動課金
- 連絡先、写真ライブラリ全体、メッセージ、閲覧履歴へのアクセス
- 本人との採点一致率の表示

## 5. 推奨アーキテクチャ

現状の `game.js` はプロトタイプとして維持し、本実装では以下へ分割する。

```text
src/
  app/
    state-machine.ts       画面遷移とセッション状態
    permissions.ts         カメラ・マイク・センサー権限
  media/
    camera.ts              撮影とカメラ切替
    face-guard.ts          人物検出と警告
    openverse.ts           画像検索・フィルター・帰属表示
    offline-generator.ts   オフライン教材生成
  speech/
    recognition.ts         音声認識アダプター
    transcript.ts          回答正規化
  training/
    prompts.ts             お題データと難易度
    features.ts            回答特徴量
    coaches.ts             10評価軸とウェイト
    scorer.ts              再現可能な採点
    tips.ts                改善ヒント
    progress.ts            再挑戦比較
  storage/
    preferences.ts         端末設定
    history.ts             練習履歴
    migrations.ts          ローカルデータ移行
  ui/
    screens/               各画面
    components/            共通部品
  sw/
    service-worker.ts      オフライン制御
```

### 状態遷移

```text
WELCOME
  → SOURCE_SELECT
  → CAPTURED
  → PROMPT_READY
  → LISTENING
  → ANALYZING
  → RESULT
  → RETRY_LISTENING
  → COMPARISON
  → SOURCE_SELECT
```

DOMの表示・非表示を個別に操作せず、状態機械から画面を描画する。

## 6. データモデル

```ts
type TrainingAttempt = {
  id: string;
  createdAt: string;
  promptId: string;
  promptText: string;
  sourceType: "camera" | "openverse" | "generated";
  answerText: string;
  responseMs: number;
  featureVersion: string;
  scoringVersion: string;
  features: Record<string, number>;
  coachScores: Array<{ coachId: string; score: number }>;
  totalScore: number;
  targetSkill: string;
  retryOf?: string;
  scoreDelta?: number;
};
```

写真と音声の原データは `TrainingAttempt` に含めない。Openverse画像の場合は、必要に応じて原典URLとライセンス情報のみを一時表示する。

## 7. 採点モデル

### 現行版

12軸（お題適合、明瞭さ、簡潔さ、新鮮さ、意外性、ずらし、具体性、展開、言葉の着地、写真接続、瞬発性、切れ味）を決定的なルールで計算する。10視点の得点は共通基準60%と人物別基準40%を基礎にし、人物別部分を資料確信度で共通基準へ縮約する。

設定は `research/judge-criteria.json` から同一オリジンで読み込み、厳格なスキーマ検証後だけ反映する。通信失敗時は最後の検証済み設定、次に内蔵設定へフォールバックする。

### 実装済み

- 乱数を完全に排除し、採点を再現可能にする。
- `scoringVersion` と `featureVersion` を全回答へ保存する。
- 各得点の計算根拠を人間が読める形で返す。
- 画像の明暗・色・密度・対称性と回答語彙の接続を評価する。
- 「役立つ／微妙」は次のヒント優先順位だけへ反映し、標準得点は変更しない。
- 研究設定、端末内ヒント評価、採点版を分離する。

### コーチID

画面表示名と内部IDを分離する。

```text
idea       発想
edge       毒と短さ
structure  構造と速度
observation 観察
oddity     異物感
finish     完成度
selection  選択
scene      情景
concept    設定
wording    言葉
```

内部IDに実在人物名を使わない。

## 8. 個人最適化

初期MVPでは採点ウェイトを自動変更しない。まず以下を記録する。

- ヒントが役立ったか
- 再挑戦したか
- 再挑戦でどの特徴が変化したか
- 得点差
- 回答時間差

十分なローカルデータが蓄積した後、端末内で次の調整を行う。

- 提示するヒントの優先順位
- お題難易度
- 制限時間
- 苦手評価軸の出題頻度

コーチの人格を学習するのではなく、ユーザーに合う練習順序を学習する。

## 9. プライバシー境界

- 権限は必要になった瞬間に個別要求する。
- アプリ開始時に全権限を一括要求しない。
- 位置情報は初期MVPから外すことを推奨する。使用する場合も整数緯度まで丸める。
- カメラ画像は採点のため外部送信しない。
- 音声はブラウザ音声認識実装による外部処理の可能性をプライバシー表示へ明記する。
- センサー情報はお題生成後に破棄する。
- 練習履歴は端末内保存を初期値とする。
- クラウド同期は明示的オプトインとする。

## 10. 現行プロトタイプの技術的課題

- `game.js` が単一ファイルで、UI・権限・採点・データ取得をすべて担当している。
- CSSが圧縮された単一行中心で保守しにくい。
- Web Speech APIは端末・ブラウザ差が大きい。
- FaceDetector APIは対応範囲が限定され、人物除外の保証にはならない。
- Openverseの人物除外はタグ文字列フィルターのみで、画像内容を保証しない。
- Openverseのライセンス情報は原典での確認が必要。
- Google Fontsは完全オフライン時に取得できない。
- Service Workerのフォールバック対象をHTMLナビゲーションに限定する必要がある。
- 現在のフィードバックは採点へ反映されない。
- 現在の採点は画像内容を理解していない。
- 既存のテストプレイ動画と録画フレームは製品ビルドから除外する。

## 11. 実装順序

1. TypeScript化と状態機械の導入
2. 採点エンジンを純粋関数として分離
3. ユニットテスト追加
4. カメラ・音声・画像取得をアダプター化
5. IndexedDBによる練習履歴
6. 再挑戦比較と成長表示
7. 権限説明とプライバシー画面
8. Service Worker再設計
9. iOS Safari・Android Chrome実機検証
10. PWA公開

## 12. 完了条件

- 1回目回答から再挑戦比較まで中断なく完走できる。
- 同じ回答・同じ応答時間・同じ採点バージョンで同じ点数になる。
- 生の写真と音声が永続保存されない。
- 実名が製品コード、画面、ログ、データモデルに存在しない。
- オフラインで教材生成、回答、採点、再挑戦が完走できる。
- 権限拒否後も利用可能な経路がある。
- 10評価軸すべての計算根拠を確認できる。
